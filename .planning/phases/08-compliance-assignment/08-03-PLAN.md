  <action>
Create new file backend/src/routes/worker-assignments.js with worker-facing assignment endpoints.

Endpoint: PATCH /api/worker-assignments/:assignmentId

Auth: Worker OTP JWT (Phase 4 worker auth)
Rate limit: standard (100/15min)

Request body:
{
  action: 'confirm' | 'decline',
  reason?: string // Optional, max 200 chars, only for decline
}

Response on success (HTTP 200):
{
  id: string,
  shiftId: string,
  workerId: string,
  workerConfirmation: 'confirmed' | 'declined',
  workerNote: string | null,
  shift: {
    id: string,
    facilityName: string,
    shiftDate: string,
    startTime: string,
    endTime: string,
    role: string
  }
}

Logic:

1. Validate request:
   - assignmentId is valid cuid
   - action is 'confirm' or 'decline'
   - If action === 'decline': reason max 200 chars
   - User authenticated (worker JWT)

2. Fetch assignment:
   - Include shift relation
   - Verify assignment.workerId === req.user.workerId (worker can only confirm their own)
   - Verify assignment.workerConfirmation === 'pending' (reject if already decided)

3. Update assignment:
   - if confirm: data: { workerConfirmation: 'confirmed', workerNote: null }
   - if decline: data: { workerConfirmation: 'declined', workerNote: reason || null }

4. Audit logging:
   - Create AuditLog entry with:
     - action: 'shift.assignment-confirmed' or 'shift.assignment-declined'
     - entity: 'ShiftAssignment'
     - metadata: { workerId, shiftId, action, reason (if decline) }

5. Return updated assignment with shift details

Error handling:
- Assignment not found: 404
- Worker mismatch: 403
- Already confirmed/declined: 400
- Invalid action: 400
- Reason > 200 chars: 400

Per R-SA-05: Worker can confirm/decline with optional reason. All actions audited.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Worker → API | Worker JWT (Phase 4 auth) required; worker can only confirm/decline their own assignments |
| API → Database | Validated workerId ensures worker can only modify their own row |
| Audit Log | Immutable record of confirm/decline actions with worker identity |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-08-11 | Spoofing | Worker confirms another's shift | Mitigate | Worker JWT verified (Phase 4). Endpoint checks assignment.workerId === req.user.workerId. Returns 403 if mismatch. |
| T-08-12 | Repudiation | Worker denies declining shift | Mitigate | Every confirm/decline logged to AuditLog with action, timestamp, worker identity. Immutable audit trail. |
| T-08-13 | Information Disclosure | Reason text in decline | Accept | Reason is worker's own input (not PII extracted from system). Stored in workerNote, visible to coordinator only (not in public API). |
| T-08-14 | Tampering | Reason modified after decline | Accept | workerNote is immutable after creation (SQL: no update triggers on this field). Audit trail shows original decline action. |

All mitigations aligned with Phase 4 worker auth and audit-ready design.

</threat_model>
