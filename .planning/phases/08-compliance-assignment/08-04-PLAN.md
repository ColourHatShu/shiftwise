Extend backend/src/cronService.js with daily compliance re-validation job.

Per R-SA-09: Cron job runs daily at 08:00 UTC. Re-validates all incomplete assignments. Flags at-risk assignments.

Job logic:

1. Schedule:
   - Run daily at 08:00 UTC (extend existing cron pattern from Phase 1)
   - Use node-cron or similar library

2. Re-validate all assignments:
   - Fetch all ShiftAssignment rows where workerConfirmation === 'pending' OR 'confirmed' (incomplete)
   - Filter: assignedAt > (today - 30 days) and shift.shiftDate >= today (upcoming shifts only)
   - For each assignment:
     a. Call validateComplianceAtTime(assignment.workerId, assignment.shiftId, assignment.agencyId)
     b. Compare new validation against original complianceSnapshot
     c. If status changed from 'compliant' to 'non-compliant':
        - Mark assignment as 'at-risk' (add field to ShiftAssignment, or update complianceCheckDetails)
        - Create coordinator alert (send email or notification)
        - Log to AuditLog: action: 'shift.compliance-revalidated', metadata: { workerId, shiftId, previousStatus, newStatus, reason }

3. Coordinator Alert Email (if assignment became non-compliant):
   - Recipients: all OWNER/ADMIN users for agency
   - Subject: "Alert: {Worker} assignment at-risk for {Shift} ({Reason})"
   - Body:
     - Worker name
     - Shift: facility, date, time
     - Reason: e.g., "Right to Work expired 2026-03-15"
     - Actions: Remove assignment, Extend deadline, Re-assign options
   - Use existing email pattern (Resend, FailedAlert on failure)

4. Error handling:
   - If validation fails: log to Sentry, continue to next assignment (don't break loop)
   - If email send fails: create FailedAlert for retry
   - Log completion: "Re-validation complete: X assignments checked, Y flagged at-risk"

5. Idempotency:
   - Only flag as at-risk once (check if already flagged)
   - Don't send alert twice for same condition

Performance:
- For <1000 shifts: completes in <30 seconds
- Use batch queries where possible
- Log to Sentry if job exceeds 60 seconds

Per R-SA-09: Daily re-validation, flag at-risk assignments, send coordinator alerts.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Coordinator → API | Shift templates CRUD, only OWNER/ADMIN authorized |
| Email system → Worker | Email contains worker's assigned shift info (no PII leakage) |
| Cron → Database | Re-validation reads all assignments, updates at-risk flags |
| Dashboard → Coordinator | Read-only view, shows aggregated metrics |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-08-15 | Tampering | Template definition modified | Mitigate | OWNER/ADMIN role check on POST/PUT/DELETE. Coordinator cannot delete templates in use. Log all changes to AuditLog. |
| T-08-16 | Spoofing | Fake "at-risk" alerts sent to coordinator | Accept | Alerts sent from system email (coordinator trusts source). Cron job is internal, not external API. Low risk. |
| T-08-17 | Information Disclosure | Email contains shift location/facility name | Accept | Facility name is not PII, worker is assigned to shift. Email recipient is worker (knows facility already). |
| T-08-18 | Denial of Service | Cron job re-validates 10000 assignments | Mitigate | Job timeout 60 sec. If exceeds timeout, logged to Sentry and next scheduled run handles remainder. Batch processing limits memory. |

All mitigations aligned with phase 1 auth and audit-ready design.

</threat_model>
