  <files>backend/src/lib/compliance-assignment.js</files>
  <action>
Create new file backend/src/lib/compliance-assignment.js that handles all compliance checking logic for assignments.

Required exports:

1. validateComplianceAtTime(workerId, shiftId, agencyId)
   Input: workerId, shiftId, agencyId
   Output: { isCompliant: boolean, reason?: string, snapshot: { documents, score, status, capturedAt, notes } }
   
   Logic:
   a. Fetch worker's ComplianceDocument rows from DB (scoped by agencyId, workerId)
   b. Fetch shift to get requiredDocuments (if shift has specific requirements; fall back to agency defaults)
   c. For each required document type:
      - Check if ComplianceDocument exists with approvalStatus = "APPROVED"
      - If missing: add to reasons array, set isCompliant = false
      - If exists and has expiryDate: check if expired (today > expiryDate), add to reasons if expired, set isCompliant = false
   d. Calculate compliance score reusing Phase 5's calculateComplianceScore(workerId, agencyId) or inline the formula:
      score = (completed_required / total_required) * 100
      where completed_required = count of required docs that are approved and not expired
      total_required = count of required document types
   e. Build immutable snapshot JSON:
      {
        documents: [array of { documentTypeId, approvalStatus, expiryDate, capturedAt: now() }],
        complianceScore: score,
        status: isCompliant ? "compliant" : "non-compliant",
        capturedAt: new Date().toISOString(),
        notes: reasons.join("; ") or null
      }
   f. Return { isCompliant, reason: notes joined with "; ", snapshot }

   Per R-SA-01: Reasons must be specific: "Missing {docType}", "Document expired {date}", "Not yet approved"

2. captureSnapshot(workerId, agencyId, shiftId)
   Wrapper that calls validateComplianceAtTime and returns just the snapshot object.
   Used by assign-bulk endpoint to store in complianceSnapshot column.

3. checkComplianceForShift(shiftId, agencyId)
   Returns compliance status for all current assignments on shift.
   Used for dashboard. Output: { assignedCount, compliantCount, atRiskCount, snapshot_details: [...] }

Dependencies:
- prisma.complianceDocument.findMany (fetch docs)
- prisma.shift.findUnique (get required docs)
- prisma.worker.findUnique (for context)
- calculateComplianceScore from compliance-service.js (or inline formula)
- Sentry for error logging (wrap in try/catch, log exceptions)

Error handling:
- Worker not found: throw 404
- Shift not found: throw 404
- Agency mismatch: throw 403
- DB errors: catch, log to Sentry, throw 500

Per R-SA-06: Snapshot is captured at time of assignment call; immutable after creation.

  <action>
Extend backend/src/routes/shift-assignments.js with new POST /api/shifts/:shiftId/assign-bulk endpoint.

Endpoint spec per R-SA-01:

Path: POST /api/shifts/:shiftId/assign-bulk
Auth: requireRole(['OWNER','ADMIN']) — only coordinators can bulk-assign
Rate limit: standard (100/15min)

Request body:
{
  workerIds: [string], // Array of worker IDs to assign
  assignmentType: 'automatic' | 'manual' // Optional, defaults to 'automatic'
}

Response on success (HTTP 200):
{
  assigned: [
    { workerId: string, shiftId: string, status: 'assigned', complianceScore: number },
    ...
  ],
  skipped: [
    { workerId: string, reason: string }, // e.g., "Missing DBS", "Document expired 2026-03-15"
    ...
  ],
  summary: {
    total: number,
    assigned: number,
    skipped: number,
    complianceScores: { mean: number, min: number, max: number }
  }
}

Response on error (HTTP 400/403/404/500):
{
  error: string,
  details?: any
}

Logic:

1. Validate request:
   - workerIds is array, not empty (< 100 max per R-SA-01 constraint)
   - All workerIds are strings (valid cuid format)
   - shiftId exists and belongs to caller's agency

2. Check shift capacity:
   - Fetch shift, count existing assignments
   - If (existingCount + workerIds.length) > shift.requiredCount: reject with "Shift is full" (but allow overassignment per Phase 7, or flag as warning)
   - Per R-SA-01: Concurrent assignments respect requiredCount via unique constraint fallback

3. Bulk check compliance (phase 7 pattern: validate all before creating any):
   - For each workerId:
     a. Call validateComplianceAtTime(workerId, shiftId, agencyId) from compliance-assignment.js
     b. If compliant: add to toAssign array with snapshot
     c. If not compliant: add to toSkip array with reason
   - If all fail compliance: return { assigned: [], skipped: [...] } (HTTP 200, partial success)

4. Atomic assignment transaction:
   - For each toAssign entry:
     a. Try: prisma.shiftAssignment.create({
          shiftId,
          workerId,
          agencyId,
          complianceSnapshot: snapshot,
          workerConfirmation: 'pending',
          complianceCheckPassed: true, // Legacy field for Phase 7 compat
          assignedAt: new Date()
        })
     b. Catch unique constraint violation (duplicate): add to toSkip with reason "Already assigned to this shift"
     c. Catch other DB error: rollback transaction, return 500

5. Audit logging:
   - For each successfully created assignment:
     prisma.auditLog.create({
       agencyId,
       userId: req.user.id,
       action: 'shift.assigned',
       entity: 'ShiftAssignment',
       entityId: assignment.id,
       metadata: {
         shiftId,
         workerId,
         complianceScore: snapshot.complianceScore,
         assignmentType
       },
       ipAddress: req.ip,
       userAgent: req.get('user-agent')
     })

6. Build response:
   - Collect all assigned and skipped
   - Calculate mean/min/max compliance scores for assigned workers
   - Return { assigned, skipped, summary }

Error handling:
- Shift not found: 404
- User not authorized: 403 (requireRole enforces)
- Invalid request body: 400 (Joi validation)
- DB transaction error: 500 (log to Sentry)

Per R-SA-01: Returns assignment results with skipped non-compliant workers and specific reasons.
Per Phase 7 pattern: Batch query then atomic create within transaction.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Client → API | Untrusted workerIds, assignmentType in request body |
| API → Compliance Service | Trusted internal call; compliance score trusted from internal calculation |
| API → Database | Trusted Prisma operations; unique constraint enforced at DB layer |
| Audit Log | Immutable record of all assignments; userId, timestamp, IP logged |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-08-01 | Tampering | complianceSnapshot JSON | Accept | Snapshot is immutable in database (never updated). Audit trail verifies any mutations. CQC inspector can validate snapshot against worker's actual doc state at time of assignment. |
| T-08-02 | Spoofing | workerIds in request | Mitigate | Validate workerIds format (cuid), check agencyId ownership, require OWNER/ADMIN role. Malicious user cannot inject arbitrary workerIds — only workers in their agency. |
| T-08-03 | Repudiation | Assignment audit trail | Mitigate | Every assignment logged to AuditLog with userId, timestamp, IP, action 'shift.assigned'. Coordinator cannot deny assigning worker. |
| T-08-04 | Information Disclosure | complianceSnapshot in response | Mitigate | Only OWNER/ADMIN can see /assignable-workers endpoint (role check). Snapshot contains sanitized doc types (no PII). snapshot.notes may mention doc types but not document numbers. |
| T-08-05 | Denial of Service | Bulk assign 1000 workers | Mitigate | Rate limit: standard 100/15min on endpoint. Max 100 workers per request enforced in validation. Batch processing with timeout. |
| T-08-06 | Elevation of Privilege | VIEWER assigns workers | Mitigate | requireRole(['OWNER','ADMIN']) enforced on POST /assign-bulk. VIEWER cannot call endpoint — returns 403. |

All mitigations aligned with Phase 1 (AUTH, ALRT) and audit-ready design (AuditLog immutable, CQC inspection ready).

</threat_model>
